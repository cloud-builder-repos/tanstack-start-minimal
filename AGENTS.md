# AGENTS.md

Operating manual for the AI agent (Codex) working in this repo. **Read it fully before changing code.**

## What this project is

The **minimal golden template** for end-user sites: a **TanStack Start frontend app** (SSR + server functions), a single deploy unit, **a single `package.json` ‚Äî not a monorepo / no workspace / no internal packages**.

- Ships **without** auth or a database by default. Add them only on demand (see "Adding a database / auth" at the end).
- Package manager: **Bun**. Toolchain: **Vite** (dev / build), **TypeScript** (strict), **Tailwind CSS v4**, **Biome** (lint + format), **Vitest** (tests).
- Routing: TanStack Router file-based routes. Server-side logic: TanStack Start server functions. Host: Nitro (dev server + deployable output; preset is switchable, see the README "Deployment").

## Hard rules (non-negotiable)

1. **When a task is complete, the quality gate must be fully green**: `bun run check-types` + `bun run lint` + `bun run test`. `bun run test` is fast ‚Äî run it freely while iterating. `check-types` and `lint` are the **final** check: run them once the feature/fix is done, not after every intermediate edit. Any red step means the task is not done ‚Äî fix until green. `bun run build` is **not** a gate ‚Äî it only produces deploy output and is slow; don't run it as a routine check.
2. **No Next.js-specific APIs**: do not import `next/*` (`next/image`, `next/link`, `next/router`, ‚Ä¶), do not write `"use client"` / `"use server"` directives, do not use App Router / `getServerSideProps` and other Next conventions. This is TanStack Start ‚Äî see "Routing & server logic" below for the equivalents.
3. **Do not touch these files**:
   - `src/routeTree.gen.ts` ‚Äî generated by the TanStack Start vite plugin at dev/build; hand edits are overwritten (already gitignored).
   - `bun.lock` ‚Äî updated only by bun commands; never edit by hand.
   - the tool config in `tsconfig.json`, `biome.json`, `vite.config.ts` / `vitest.config.ts` ‚Äî the quality-gate baseline.
   - `dist/`, `.tanstack/`, `.wrangler/` ‚Äî generated directories.
4. **Do not swap tools or add parallel infrastructure**: do not introduce another package manager, lint/format tool, test framework, or a second HTTP/SSR framework; **do not turn the project into a workspace / monorepo**; do not create new top-level directories. If a capability seems missing, find it in the existing stack first.
5. **No `console.*` on production paths in new code**: remove debug logging when done.

## Common commands (run from the repo root)

```bash
bun install           # Install dependencies
bun run dev           # Local dev (Vite + Nitro), http://localhost:3000
bun run build         # Production build (Cloudflare Worker, output in dist/)
bun run check-types   # tsc --noEmit only
bun run lint          # biome check (lint + format check, no writes)
bun run format        # biome check --write (auto-fix format/lint)
bun run test          # vitest run (single run)
```

**Always invoke scripts as `bun run <script>`.** Plain `bun build` and `bun test` run Bun's own bundler / test runner, **not** these package.json scripts ‚Äî using them by mistake will not run Vite or Vitest.

Note: `bun run check-types` (tsc) needs a fresh `src/routeTree.gen.ts`. The dev server (`bun run dev`) regenerates it automatically as you add/remove route files, so keep `bun run dev` running while you work and `check-types` will see the current routes. (A `build` also regenerates it, but don't run a build just for this.)

## Directory layout

```
src/
  routes/            # File-based routes (pages). Route files only
    __root.tsx       # Root document (<html>/<head>/<body>, global <head> meta, style mount)
    index.tsx        # Home "/"
  components/         # Shared React components (PascalCase file name + export)
  server/            # Server functions (createServerFn)
  lib/               # Pure logic / isomorphic utilities (unit-testable); *.test.ts next to source
  styles.css         # Tailwind entry (@import "tailwindcss")
public/              # Static assets (robots.txt, etc.), served at their path
```

## Routing & server logic

Pick by scenario, top priority first:

1. **Pages ‚Üí file-based routes**: add a file under `src/routes/` following TanStack Router conventions (`index.tsx` = `/`, `about.tsx` = `/about`, `$param` = dynamic segment, `_layout/` = layout group). A new page = a new file + `createFileRoute("<path>")({ component, loader? })`. `routeTree.gen.ts` is updated by the build ‚Äî **do not hand-write it**.
2. **Logic that must run on the server (files / secrets / backend calls) ‚Üí server function**: put it in `src/server/<name>.ts` with `createServerFn({ method }).inputValidator(fn).handler(fn)` (see `src/server/hello.ts`). Call it from a route `loader` for SSR data fetching (see `src/routes/index.tsx`), or from a client event. **Do not hand-write fetch to call your own backend** ‚Äî a server function is same-origin RPC.
3. **Endpoints that must control the raw Request/Response (webhooks, third-party callbacks) ‚Üí API route**: use `createFileRoute`'s `server.handlers` (GET/POST/‚Ä¶) under `src/routes/`. Not needed for the default case ‚Äî prefer the first two.

## Component & styling conventions

- **Tailwind CSS v4**, utility-first: styles go in JSX `className`. **Do not** create extra `.css` files, CSS Modules, CSS-in-JS, or inline `style`. There is exactly one global CSS file: `src/styles.css` (just `@import "tailwindcss";`).
- Shared components go in `src/components/` (PascalCase file name and export, e.g. `Card.tsx`). See `src/components/Card.tsx`.
- Keep components presentational; push testable business/formatting logic down into `src/lib/` and unit-test it.

## Testing conventions

- Framework: **Vitest**. Test files sit next to their source, named `*.test.ts` / `*.test.tsx` (see `src/lib/greeting.test.ts`).
- The test runtime does **not** load the tanstackStart/cloudflare plugins (see `vitest.config.ts`), so prefer testing **pure logic / pure components**; extract logic out of routes / server functions into `src/lib/` to make it testable.
- New business logic must come with tests; when fixing a bug, write the reproducing test first.

## Adding dependencies

- `bun add <dep>` (runtime) / `bun add -d <dep>` (dev). Prefer reusing existing dependencies; add nothing you can avoid (minimalism is this template's value).
- Do not pre-install things "in case we need them later".

## Adding a database / auth (on-demand platform capability)

This template deliberately ships **without** a database, auth, or an env-validation library. These are **on-demand platform capabilities**, not part of the base.

- When the project genuinely needs persistence or login, **follow the `add-database` skill** (provided by the platform). The platform's database is **Cloudflare D1** ‚Äî a Worker binding, **not** a connection string.
- Do **not** add Postgres, a `DATABASE_URL`, or any raw connection string; do **not** provision cloud resources or put secrets in the project. The platform provisions and binds the database at deploy time.
- Before requesting a database, check whether a server function + static/external APIs already solves the need ‚Äî reach for a database only when it is truly required.
