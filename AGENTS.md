# AGENTS.md

Operating manual for the AI agent (Codex) working in this repo. **Read it fully before changing code.**

## What this project is

The **minimal golden template** for end-user sites: a **TanStack Start frontend app** (SSR + server functions), a single deploy unit, **a single `package.json` — not a monorepo / no workspace / no internal packages**.

- Ships **without** auth or a database by default. Add them only on demand (see "Adding a database / auth" at the end).
- Toolchain: **Vite+** (the `vp` CLI) — one tool wrapping Vite + Rolldown (dev / build), Vitest (tests), Oxlint (lint), Oxfmt (format), and type-checking. Package manager: **Bun** (detected and managed by `vp`). Language: **TypeScript** (strict). Styling: **Tailwind CSS v4**.
- UI: **shadcn/ui** (Base UI primitives + Tailwind), configured via `components.json`, imported through the `@/*` alias. See "Component & styling conventions".
- Routing: TanStack Router file-based routes. Server-side logic: TanStack Start server functions. Host: Nitro (dev server + deployable output; preset is switchable, see the README "Deployment").

## Hard rules (non-negotiable)

1. **When a task is complete, the quality gate must be fully green**: `vp check` (format + lint + type-check) and `vp test`. `vp test` is fast — run it freely while iterating. `vp check` is the **final** check: run it once the feature/fix is done, not after every intermediate edit. Any red step means the task is not done — fix until green. `vp build` is **not** a gate — it only produces deploy output and is slow; don't run it as a routine check.
2. **No Next.js-specific APIs**: do not import `next/*` (`next/image`, `next/link`, `next/router`, …), do not write `"use client"` / `"use server"` directives, do not use App Router / `getServerSideProps` and other Next conventions. This is TanStack Start — see "Routing & server logic" below for the equivalents.
3. **Do not touch these files**:
   - `src/routeTree.gen.ts` — generated by the TanStack Start vite plugin at dev/build; hand edits are overwritten (already gitignored).
   - `bun.lock` — updated only by `vp` / bun; never edit by hand.
   - `components.json` — shadcn/ui config; managed by the shadcn CLI, not by hand.
   - the tool config in `tsconfig.json`, `vite.config.ts`, `vitest.config.ts` — the quality-gate baseline. `vite.config.ts` holds the Vite+ config: the `lint` (Oxlint) and `fmt` (Oxfmt) blocks, and the app plugins wrapped in `lazyPlugins(() => [...])` (leave that wrapper — Vite+ needs it for check/lint/fmt). The `@/*` → `./src/*` path alias in `tsconfig.json` is required by shadcn/ui — leave it in place.
   - `dist/`, `.tanstack/`, `.wrangler/` — generated directories.
4. **Do not swap tools or add parallel infrastructure**: the toolchain is **Vite+** — go through `vp`. Do not add a standalone package manager (npm/pnpm/yarn), linter, formatter, test runner, or bundler (ESLint, Prettier, Biome, raw Vite/Vitest), or a second HTTP/SSR framework; **the UI layer is shadcn/ui (Base UI + Tailwind) — do not add a different component library** (MUI, Chakra, Ant, Mantine, or Radix directly); **do not turn the project into a workspace / monorepo**; do not create new top-level directories. If a capability seems missing, find it in the existing stack first.
5. **No `console.*` on production paths in new code**: remove debug logging when done.

## Common commands (run from the repo root)

```bash
vp install     # Install dependencies (Bun under the hood)
vp dev         # Local dev (Vite + Nitro), http://localhost:3000
vp build       # Production build (Cloudflare Worker, output in dist/)
vp check       # Format + lint + type-check — the quality gate
vp test run    # Run tests once (Vitest)
vp lint        # Lint only (Oxlint)
vp fmt         # Format in place (Oxfmt)
```

`vp` owns the package manager (Bun) and Node version — don't call `bun`/`npm` directly for install or scripts; use `vp`. The `package.json` scripts (`dev`, `build`, `check`, `test`, …) are thin wrappers that call `vp`, so `bun run <script>` still works if you prefer.

Note: `vp check`'s type-checker needs a fresh `src/routeTree.gen.ts`. The dev server (`vp dev`) regenerates it automatically as you add/remove route files, so keep `vp dev` running while you work and `vp check` will see the current routes. (A `vp build` also regenerates it, but don't run a build just for this.)

## Directory layout

```
src/
  routes/            # File-based routes (pages). Route files only
    __root.tsx       # Root document (<html>/<head>/<body>, global <head> meta, style mount)
    index.tsx        # Home "/"
  components/         # Shared React components (PascalCase file name + export)
    ui/              # shadcn/ui primitives — generated by the shadcn CLI, don't hand-write
  server/            # Server functions (createServerFn)
  lib/               # Pure logic / isomorphic utilities (unit-testable); *.test.ts next to source
                     #   utils.ts holds cn() (shadcn class helper)
  styles.css         # Tailwind entry + shadcn theme (CSS variables, @theme inline, :root/.dark)
public/              # Static assets (robots.txt, etc.), served at their path
```

## Routing & server logic

Pick by scenario, top priority first:

1. **Pages → file-based routes**: add a file under `src/routes/` following TanStack Router conventions (`index.tsx` = `/`, `about.tsx` = `/about`, `$param` = dynamic segment, `_layout/` = layout group). A new page = a new file + `createFileRoute("<path>")({ component, loader? })`. `routeTree.gen.ts` is updated by the build — **do not hand-write it**.
2. **Logic that must run on the server (files / secrets / backend calls) → server function**: put it in `src/server/<name>.ts` with `createServerFn({ method }).validator(fn).handler(fn)` (see `src/server/hello.ts`). Call it from a route `loader` for SSR data fetching (see `src/routes/index.tsx`), or from a client event. **Do not hand-write fetch to call your own backend** — a server function is same-origin RPC.
3. **Endpoints that must control the raw Request/Response (webhooks, third-party callbacks) → API route**: use `createFileRoute`'s `server.handlers` (GET/POST/…) under `src/routes/`. Not needed for the default case — prefer the first two.

## Component & styling conventions

- **Tailwind CSS v4**, utility-first: styles go in JSX `className`. **Do not** create extra `.css` files, CSS Modules, CSS-in-JS, or inline `style`. There is exactly one global CSS file: `src/styles.css` (Tailwind entry + the shadcn theme). Style with the theme tokens (`bg-background`, `text-foreground`, `border-border`, …) rather than hard-coded colors; don't hand-edit the generated theme block.
- **shadcn/ui**: add primitives with `bunx shadcn@latest add <name>` — they land in `src/components/ui/` and are **generated, not hand-written** (edit only to customize intentionally). Compose your own components in `src/components/` from these primitives. Merge classes with `cn()` from `@/lib/utils`. Import via the `@/*` alias (`@/components/ui/button`, `@/lib/utils`).
- Shared app components go in `src/components/` (PascalCase file name and export, e.g. `Card.tsx`). See `src/components/Card.tsx`.
- Keep components presentational; push testable business/formatting logic down into `src/lib/` and unit-test it.

## Testing conventions

- Framework: **Vitest**, run via `vp test`. Test files sit next to their source, named `*.test.ts` / `*.test.tsx` (see `src/lib/greeting.test.ts`). Import test APIs from **`vite-plus/test`** (not `vitest`): `import { describe, expect, it } from "vite-plus/test"`.
- The test runtime does **not** load the tanstackStart/cloudflare plugins (see `vitest.config.ts`), so prefer testing **pure logic / pure components**; extract logic out of routes / server functions into `src/lib/` to make it testable.
- New business logic must come with tests; when fixing a bug, write the reproducing test first.

## Adding dependencies

- `vp add <dep>` (runtime) / `vp add -D <dep>` (dev). Prefer reusing existing dependencies; add nothing you can avoid (minimalism is this template's value).
- Do not pre-install things "in case we need them later".

## Adding a database / auth (on-demand platform capability)

This template deliberately ships **without** a database, auth, or an env-validation library. These are **on-demand platform capabilities**, not part of the base.

- When the project genuinely needs persistence or login, **follow the `add-database` skill** (provided by the platform). The platform's database is **Cloudflare D1** — a Worker binding, **not** a connection string.
- Do **not** add Postgres, a `DATABASE_URL`, or any raw connection string; do **not** provision cloud resources or put secrets in the project. The platform provisions and binds the database at deploy time.
- Before requesting a database, check whether a server function + static/external APIs already solves the need — reach for a database only when it is truly required.
